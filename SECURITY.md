# Security Policy — سياسة الأمان

## الإبلاغ عن الثغرات (Reporting a Vulnerability)

أمان هذا النظام أولوية قصوى — فهو يدير وصولاً عن بُعد إلى أجهزة العملاء.

**لا تفتح Issue عاماً للثغرات الأمنية.** بدلاً من ذلك:

- راسل الفريق مباشرة عبر البريد: **security@al-abbasi.soft** (أو القناة الداخلية المعتمدة).
- صف الثغرة، خطوات إعادة إنتاجها، والأثر المحتمل.
- سنردّ خلال **72 ساعة** ونعمل على إصلاح عاجل للثغرات الحرجة.

## نطاق الحساسية (Sensitive Scope)

- مفاتيح SSH الخاصة (تُخزَّن مشفّرة AES-256-GCM، تُسلَّم للأدمن فقط عبر HTTPS).
- مفتاح تشفير قاعدة البيانات (`MOTECH_KEY_ENCRYPTION_KEY`).
- توكنات NetBird (PAT) ورموز التفعيل (one-time).
- شهادات توقيع الكود (المفاتيح الخاصة لا تُرفع أبداً).

## ممارسات الأمان المطبّقة (Implemented Controls)

| Control | Status |
|---------|--------|
| تشفير المفاتيح الخاصة at rest (AES-256-GCM) | ✅ |
| HTTPS فقط (Caddy، HTTP→308→HTTPS) | ✅ |
| JWT منفصل للأدمن والوكيل (kind claim) | ✅ |
| رموز تفعيل لمرة واحدة (hashed في DB) | ✅ |
| Backend-owned keys (الوكيل لا يولّد/يحمل مفاتيح) | ✅ |
| تدوير المفاتيح + حذف القديم نهائياً | ✅ |
| سجل نشاط كامل لكل وصول للمفاتيح | ✅ |
| توقيع Authenticode للملفات التنفيذية | ✅ |
| لا secrets في الكود أو git history | ✅ |

## ما يجب ألا يُرفع أبداً إلى Git

`.env` · `*.key` · `*.pem` · `*.pfx` · `ca.key` · أي PAT/token/password.
(محميّة عبر `.gitignore`؛ راجعها قبل أي commit.)
